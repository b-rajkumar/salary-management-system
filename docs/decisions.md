# Decisions

**No auth.** Skipping the auth sectoin as this is an assignment where the main focus is on the core functionality

**Single `employees` table.** Real HR systems split department, country, and job title into reference tables, track salary history, and keep an audit log. Skipped here — the gaps are called out in the PRD instead of being half-built.

**MUI for the UI.** The 10k-row table is the hard part of the frontend. DataGrid handles virtualized scroll and server-side paging, sort, and filter out of the box. 

**Per-country currency.** Salary is stored as a whole number in the local currency of the employee's country. The country to currency mapping lives as a frozen map in `shared/` — derived at display time, never stored on the employee row. No FX conversion anywhere; what HR enters is what gets aggregated and shown.

**better-sqlite3 with kysely.** For this particular application I am going with better-sqlite3 and kysely for forming queries.

**Country change in the edit form clears the salary field.** When HR changes an employee's country in the edit dialog, the salary input is auto-cleared and an inline helper prompts HR to re-enter the value in the new currency. Salary is stored in the employee's local currency (derived from the country); a country change silently re-interprets the same number in a different currency, which is the highest-risk edit available without an audit log. Forcing re-entry trades a small amount of friction for an unambiguous data record. The alternative — a yellow warning that leaves the value in place — relies on HR noticing, which is the failure mode we are designing against.

**Hard delete with a simple confirm modal.** When HR deletes an employee, the row is removed from `employees` with no soft-delete column, no audit log, and no undo. The confirmation is a standard MUI modal showing the employee's name, email, and country (the three identifiers that disambiguate a row in the grid); Cancel is the default-focused button and Delete is red. Two stronger alternatives were considered and rejected: typed-name confirmation (overkill for a single trusted operator) and undo-via-snackbar (requires either a soft-delete column — explicitly out of scope per CLAUDE.md YAGNI — or a client-side stash + recreate flow, which adds real complexity without solving the underlying problem). The chosen design follows the existing assumption that the deployed instance is operated by a single trusted user, and that a deliberate confirm modal is sufficient friction against a stray click.

**Department breakdown lives inside the country response.** Rather than a second endpoint (`/api/insights/country/:country/departments`), the per-department `[{ department, headcount, avgSalary }]` list is returned inline on `/api/insights/country/:country`. Both pieces share the same `country` filter and are always rendered together on the page — splitting them would mean two round trips, two loading states, and no independent cacheability gain at this scale. The response is ~10 small rows larger; negligible at 10k employees.

**Separate `InsightsRepository`, not methods on `EmployeesRepository`.** Insight queries are aggregate / group-by / order-statistic operations — fundamentally different from CRUD. Co-locating them in `EmployeesRepository` would muddle the contract of a class whose other methods are row-shaped. The new `InsightsRepository` parallels the `InsightsController` / `InsightsService` layering and keeps each repository's purpose readable from its filename. (An earlier draft of `engineering-design.md §6` sketched these queries on `EmployeesRepository`; that flow is rewritten in the same change.)

**Role insights are a filter on the country view, not a parallel report.** The Insights page started with two stacked sections (FR-5 country view, FR-6 role view). The role view was reframed as a *filter* on the country view: one set of stat cards that update when a role is picked, departments table hidden in role mode, and an explicit "+/-N% vs all roles in {country}" delta on the salary card. The comparison delta is the answer HR is actually looking for — showing role numbers in isolation forces them to remember the country mean and do the math. The trade-off is a non-trivial refactor of the already-shipped FR-5 component (`CountryInsightsCard` → unified `InsightsCard`), accepted because the unified surface is smaller code and a cleaner UI than two parallel cards stacked on the page.

**Job-title matching is case-insensitive.** The picker collapses casing variants (`SELECT MIN(jobTitle) ... GROUP BY LOWER(jobTitle)`) so "IT Manager" and "it manager" surface as one role; the aggregate endpoint matches with `jobTitle = ? COLLATE NOCASE`. HR thinks of these as the same role; treating them as distinct would split aggregates and clutter the picker. To honor NFR-3 the schema's `(country, jobTitle)` index was replaced with `(country, jobTitle COLLATE NOCASE)` in migration 002 — the binary-collation index cannot serve a `COLLATE NOCASE` predicate.

**Empty-data insights return 404, not 200 with nulls.** `GET /api/insights/country/:country` (and the role variant) respond `404 COUNTRY_NOT_FOUND` / `ROLE_NOT_FOUND` when no rows match — the response shape (min/max/avg salary, avg tenure) is undefined over a zero set, so a strict shared type beats nullable fields. The frontend wrapper maps the 404 to an inline "No employees in {country} yet" state; HR never sees an error banner.

**Containerize even though Render can build Node natively.** We ship a multi-stage `Dockerfile` and deploy via Render's Docker env, not Render's auto-detected Node build. The image pins the Node 20 version, the `better-sqlite3` native binary, and every transitive dep into a reproducible artifact that survives Render's native-builder changes and is portable if we ever leave Render. The cost is ~50 lines of Dockerfile and a small compose file. The native-build alternative was considered: `engines: node >=20` + `npm ci && npm run build` + `node backend/dist/server.js` works today, but offers none of the reproducibility guarantees and ties us to Render's build environment.

**Render free tier with ephemeral SQLite and seed-on-empty.** The deployed app runs on Render's free web service, which doesn't support persistent disks. Cold starts re-create the schema and re-seed 10k demo rows via the `SEED_ON_EMPTY=1` startup hook, costing ~10s on the first request after a 15-min idle. Persistence is one `disk:` block and a $7/mo Starter plan-bump away — no code change. Render's free Postgres was considered as the persistence alternative but rejected: it invalidates the SQLite/Kysely/migration decisions above, doubles operational surface, and the reviewer's experience is the same either way (a populated demo at the URL).

**Bulk CSV import (FR-7) is all-or-nothing.** A CSV upload either inserts every row or inserts none. The alternative — insert valid rows, skip duplicates with a warning — was rejected because it silently discards HR's intent: if the CSV row is the corrected version of an existing record, "skip the duplicate" preserves the stale row without telling HR which they got. Forcing HR to fix the CSV (or use `PUT /api/employees/:id` to update an existing record) keeps the data record unambiguous. The trade-off is that one bad row blocks the whole upload; mitigated by returning every row-level error in one response so HR can fix them in a single re-upload. The 500-row cap is set deliberately so the whole insert finishes in one synchronous HTTP round-trip (no async/queue infrastructure) and stays out of "re-seed the whole DB via the UI" territory — larger loads still go through the seed script. Matches the posture of the "country change clears salary" decision: when the data semantics are ambiguous, force a deliberate action rather than silently doing the wrong thing.

**Email is lowercased at the Zod boundary.** `employeeCreateSchema.email` applies `.trim().toLowerCase()` so the canonical form reaches the service layer. The `employees.email` column has SQLite's default binary collation, so without this both `Asha@x.com` and `asha@x.com` would slip past the `UNIQUE` constraint as distinct values. Normalizing at the schema means the single-create endpoint, the bulk endpoint, and the bulk endpoint's pre-check `SELECT email WHERE email IN (...)` all agree on case — and the case-sensitive index correctly answers a case-insensitive question. The alternative (lowercasing inside each repository write + casting `LOWER(email)` in the pre-check) would have required a function-based index or a table scan to use the unique index.
